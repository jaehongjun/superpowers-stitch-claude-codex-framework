Reusable Operating Framework

Superpowers + Claude Design + Claude + Codex

특정 서비스용 플레이북이 아니라, 여러 프로젝트에서 반복적으로 가져다 쓸 수 있는 범용 개발 운영 문서다. 문제 정의를 먼저 좁히고, 화면 기준점을 확정한 뒤, 계획과 구현을 분리해 시행착오를 줄이는 데 초점을 둔다.

워크플로우 보기 프롬프트 템플릿 보기
문제 정의 초반 범위를 짧고 정확하게 좁힌다.
화면 기준점 Claude Design을 기본 기준점으로 잡고, Stitch는 보조 옵션으로 둔다.
계획 문서화 Superpowers로 구멍과 태스크를 구조화한다.
실제 구현 Codex가 코드 구조와 검증 중심으로 마무리한다.

문서 목적

이 프레임워크는 소비자용 웹 서비스, SaaS 대시보드, 내부 업무툴, 여행·커머스·예약 서비스, 콘텐츠·추천 서비스, MVP 웹앱까지 폭넓게 적용할 수 있다.

정의

초반 문제 정의를 짧고 정확하게 좁히고, MVP 범위를 선명하게 만든다.

디자인 기준

화면의 첫 기준점을 먼저 잡아서 구현 품질이 흔들리지 않게 한다.

운영 규칙

worktree, UI 제약, 역할 분리, 추론 강도까지 한 흐름으로 고정한다.

전체 워크플로우

핵심은 문제 정의는 Codex /plan으로, 기획 구조화는 Superpowers로, 기준 화면은 Claude Design으로, 구현 가능성 검수와 구조 정제는 Claude로, 실제 구현은 Codex로 분리하는 것이다.

1
Codex /plan 인터뷰

핵심 문제, 주요 사용자, MVP 범위와 비범위를 먼저 확정한다.

2
Superpowers Brainstorming

빠진 요구사항, UX 리스크, edge case, 흐름상 빈 구멍을 점검한다.

3
Claude Design 기준 디자인

기준 화면과 프로토타입을 만들고, 이후 구현의 시각적 기준점으로 삼는다.

4
Claude 구현 가능성 점검 + 구조 정제

디자인 결과를 검수하며 구현 가능성, 정보 위계, CTA, 카드 구조를 제품 수준으로 다듬는다.

5
Superpowers Writing Plans

기술스택, UI 제약, API 흐름, 컴포넌트 후보를 구현 계획으로 변환한다.

6
Superpowers Git Worktrees

기능 단위를 안전하게 쪼개고 merge, env, install 규칙까지 고정한다.

7
Codex 코드베이스 분석

Writing Plans와 Claude 결과를 현재 저장소 구조에 매핑하고 진입점과 영향 범위를 찾는다.

8
Codex 구현 계획 고정

최소 수정 원칙, 재사용 컴포넌트 범위, API 연동 계획을 실제 수정 전 잠근다.

9
Codex 실제 구현

새 디자인 제안보다 기준점 재현을 우선하며 실제 코드 작성과 검증을 진행한다.

10
Claude 최종 UI 정제

감성은 유지하면서 cheap한 느낌만 걷어내고 spacing, hierarchy, CTA 밀도를 정교하게 마무리한다.

11
필요 시 Codex 코드리뷰

회귀 위험, 상태 처리 누락, edge case, 복잡도, 테스트 부족을 리뷰하고 필요한 부분만 최소 수정한다.

역할 분담

디자인 창작과 구현을 분리하고, 각 도구가 가장 잘하는 지점만 맡긴다.

Codex

  • 문제 정의 인터뷰
  • 코드베이스 분석
  • 구현 계획 고정
  • 실제 코드 작성
  • 필요 시 마지막 코드리뷰
  • 테스트와 검증

Superpowers

  • 브레인스토밍
  • 구현 계획 문서화
  • 태스크 분해
  • worktree 전략 정리

Claude Design

  • 기준 화면과 프로토타입 생성
  • 레이아웃과 섹션 기준점 생성
  • 가능하면 handoff 또는 export 결과 확보

Claude

  • 구현 가능성 점검
  • 정보 위계와 카드 구조 정리
  • cheap한 AI 느낌을 세련되게 정제
  • 최종 UI 완성도 보정

단계별 운영 방식

기술스택과 UI 제약은 너무 일찍 고정하지 않고, Writing Plans에서 본격 반영한 뒤 구현 단계에서 유지한다. 디자인 출력 후에는 바로 계획으로 넘기지 않고, Claude가 먼저 구현 가능한 구조인지 검수한다.

1단계. Codex /plan 인터뷰

문제 정의, 사용자, 핵심 가치, MVP 범위와 결과 단위를 먼저 확정한다.

  • 기능 목록과 화면 구조는 아직 만들지 않는다.
  • 기술스택은 깊게 논의하지 않고 제품 제약만 말한다.
  • 추론 강도: 심화추론
2단계. Superpowers Brainstorming

빠진 요구사항과 UX 리스크, edge case, 상태 처리 구멍을 찾는다.

  • 기술보다 사용자 흐름과 서비스 신뢰성에 집중한다.
  • 추천·탐색·입력·저장·관리 흐름을 점검한다.
  • 1단계에서 이미 확정한 가설을 다시 인터뷰처럼 캐묻지 않는다.
  • 출력은 질문보다 분석 결과와 보완안이 우선이어야 한다.
  • 추론 강도: 심화추론
3단계. Claude Design 기준 디자인

주요 화면의 기준 디자인과 프로토타입을 잡아 이후 구현의 시각적 기준점으로 삼는다.

  • 기본값은 Claude Design이고, Stitch는 필요할 때만 보조 옵션으로 둔다.
  • 구현 기준점으로 사용할 수 있을 만큼 완성도 있게 만든다.
  • 추론 강도: 보통
4단계. Claude 구현 가능성 점검 + 구조 정제

디자인 결과를 검수해 Writing Plans로 넘겨도 되는 구조인지 확인하고, 정보 위계와 톤을 정교하게 만든다.

  • 화면 목록, 화면 목적, 카드/상세/CTA 위계, 인터랙션 모호성을 함께 본다.
  • component 단위 분해와 empty/loading/error 정의 가능성까지 점검한다.
  • 추론 강도: 보통 ~ 약간 심화
5단계. Superpowers Writing Plans

디자인 출력이 암시한 구조를 실제 구현 가능한 제품 구조로 해석하고 확정한다.

  • 기술스택을 여기서 처음 본격적으로 확정한다.
  • route vs inline structure, component candidates, state definitions를 정리한다.
  • 브라우저 기본 input, modal 지양 같은 제약도 여기서 잠근다.
  • 추론 강도: 보통, 구현 전략 판단은 약간 심화
6단계. Superpowers Git Worktrees

기능 단위 분리와 운영 규칙을 함께 설계해 병렬 작업 리스크를 줄인다.

  • foundation과 feature 작업을 분리할지 판단한다.
  • gitignore, env, install, merge 규칙까지 함께 정리한다.
  • 추론 강도: 보통
7단계. Codex 코드베이스 분석

Writing Plans와 Claude Design·Claude 결과를 현재 저장소 구조에 자연스럽게 매핑한다.

  • 관련 파일 후보, 라우팅 구조, 진입점, 위험한 영향 범위를 찾는다.
  • 기술 재설명보다 디자인 기준점을 현재 컴포넌트 구조로 옮기는 방식 제안이 핵심이다.
  • 추론 강도: 심화추론
8~10단계. 구현 계획 고정 · 구현 · 최종 정제

최소 수정 원칙으로 구현하고, 기준점 재현을 우선한 뒤 마지막엔 세련되게 정제한다.

  • Codex는 새 디자인 제안보다 Claude Design 기준점 재현을 우선한다.
  • Claude는 boring하거나 촌스러운 방향으로 죽이지 않고 cheap함만 걷어낸다.
  • 추론 강도: 대체로 보통
11단계. 필요 시 Codex 코드리뷰

구현 후 선택적으로 /review를 돌려 회귀 위험, 로직 실수, 상태 누락, edge case를 마지막으로 점검한다.

  • 기존 코드베이스가 크거나 여러 worktree로 나뉜 작업일수록 효과가 크다.
  • 작은 수정, 정적 퍼블리싱, 초기 프로토타입이면 생략 가능하다.
  • 추론 강도: 보통

2. Superpowers Brainstorming

brainstorming

방금 좁힌 MVP를 기준으로 구멍을 찾아줘.

중요:
- 웬만하면 나에게 추가 질문하지 말고, 합리적 가정을 두고 바로 분석할 것
- 정보가 조금 부족해도 진행 가능한 범위까지는 먼저 정리할 것
- 정말 진행이 불가능한 핵심 정보가 빠진 경우에만 마지막에 부족한 정보 3개 이하로 짧게 적을 것
- 결과는 비평만 하지 말고, 문서에 바로 붙일 수 있는 수정 제안 문구까지 포함할 것
- 이 단계에서 1단계 가설을 다시 인터뷰처럼 캐묻지 말 것
- 출력은 질문이 아니라 구멍 분석과 보완안 중심일 것

서비스 핵심 가설:
- [핵심 가치]
- [사용자 흐름 요약]
- [결과물 형태]

중점:
- 빠진 요구사항
- UX 리스크
- edge case
- 사용자가 이해하기 어려운 지점
- 필터/입력/탐색/상세/저장 구조
- empty / loading / error 상태

결과 형식:
1. 놓친 요구사항
2. UX 리스크
3. edge case
4. 상태별 처리
5. 문서에 바로 붙일 수정 제안 문구
6. 가정
7. 정말 필요할 때만 부족한 정보

5. Superpowers Writing Plans

writing-plans

디자인 출력이 암시한 구조를 실제 구현 가능한 구조로 해석하고 확정해줘.

중요:
- 무에서 구조를 새로 만들기보다 디자인 결과를 제품 구조로 번역할 것
- 디자인 구조와 실제 제품 구조가 다르면 제품 구조 기준으로 정리할 것
- 디자인 출력이 내 요구사항, 사용자 흐름, 핵심 화면 목적에 실제로 맞는지 먼저 검증할 것
- 요구사항과 맞지 않거나 정보가 부족하면 억지로 진행하지 말고 무엇이 부족한지 먼저 명확히 지적할 것
- 화면디자인의 중요한 문구, 정보 위계, CTA, 상태 흐름은 반드시 디자인 결과와 맞춰 해석할 것
- 요구사항 불일치가 크면 계획 확정으로 넘어가지 말고 확인이 필요한 항목을 먼저 정리할 것

기술 전제:
- [framework]
- [language]
- [styling]
- [state/data fetching preference]

UI 구현 제약:
- [브라우저 기본 input 지양]
- [modal/dialog 중심 흐름 지양]

부족하거나 맞지 않는 경우 형식:
1. Mismatches with requirements
2. Missing inputs
3. What must be clarified before proceeding

기술스택과 Claude Design

기술스택 1단계에서는 제품 제약만, 5단계 Writing Plans에서 본격 반영한다.
Claude Design 참조 3단계에서 생성하고 4단계 검수 후 5단계, 7단계, 9단계에서 구현 기준점으로 사용한다.
UI 제약 브라우저 기본 input, modal 중심 흐름 지양 규칙은 Writing Plans에서 확정하고 구현 직전에 다시 상기한다.

AI 슬롭 대신 정제

피해야 할 표현 AI 슬롭 제거, 과한 요소 제거, 현실적으로만 수정 같은 지시는 디자인을 죽이기 쉽다.
권장 방향 감성 유지, premium 무드 유지, cheap한 느낌만 제거, 위계와 spacing만 정교하게 다듬는다.
핵심 문장 슬롭 제거보다 세련되게 정제가 더 맞다.

최종 압축 버전

새 프로젝트를 시작할 때 빠르게 복기할 수 있는 한 줄 흐름이다.

1. Codex /plan 인터뷰
2. Superpowers Brainstorming
3. Claude Design으로 기준 디자인/프로토타입 만들기
4. Claude로 구현 가능성 점검 + 구조/톤/정보 위계 보정
5. Superpowers Writing Plans
6. Superpowers Git Worktrees
7. Codex 코드베이스 분석
8. Codex 구현 계획 고정
9. Codex 실제 구현
10. Claude로 최종 UI 정제
11. 필요 시 Codex 코드리뷰
12. 마무리 수정
Claude Design이 먼저 화면 기준점을 만들고, Claude가 그걸 구현 가능한 제품 구조로 검수하고 다듬고, Superpowers가 실행 계획으로 바꾸고, Codex가 기준점에 맞춰 구현한 뒤 마지막에 필요 시 코드리뷰까지 보는 흐름을 반복 가능한 범용 프레임워크로 사용하는 것이 이 문서의 핵심이다.
