Reusable Operating Framework

Superpowers + Stitch + Claude + Codex

특정 서비스용 플레이북이 아니라, 여러 프로젝트에서 반복적으로 가져다 쓸 수 있는 범용 개발 운영 문서다. 문제 정의를 먼저 좁히고, 화면 기준점을 확정한 뒤, 계획과 구현을 분리해 시행착오를 줄이는 데 초점을 둔다.

워크플로우 보기 프롬프트 템플릿 보기
문제 정의 초반 범위를 짧고 정확하게 좁힌다.
화면 기준점 Stitch로 먼저 구조와 분위기를 고정한다.
계획 문서화 Superpowers로 구멍과 태스크를 구조화한다.
실제 구현 Codex가 코드 구조와 검증 중심으로 마무리한다.

문서 목적

이 프레임워크는 소비자용 웹 서비스, SaaS 대시보드, 내부 업무툴, 여행·커머스·예약 서비스, 콘텐츠·추천 서비스, MVP 웹앱까지 폭넓게 적용할 수 있다.

정의

초반 문제 정의를 짧고 정확하게 좁히고, MVP 범위를 선명하게 만든다.

디자인 기준

화면의 첫 기준점을 먼저 잡아서 구현 품질이 흔들리지 않게 한다.

운영 규칙

worktree, UI 제약, 역할 분리, 추론 강도까지 한 흐름으로 고정한다.

전체 워크플로우

핵심은 문제 정의는 Codex /plan으로, 기획 구조화는 Superpowers로, 화면 기준점은 Stitch로, 제품 구조 정제는 Claude로, 실제 구현은 Codex로 분리하는 것이다.

1
Codex /plan 인터뷰

핵심 문제, 주요 사용자, MVP 범위와 비범위를 먼저 확정한다.

2
Superpowers Brainstorming

빠진 요구사항, UX 리스크, edge case, 흐름상 빈 구멍을 점검한다.

3
Stitch 기준 디자인

구현의 시각적 기준점이 되는 메인 화면과 구조를 만든다.

4
Claude 구조 정제

정보 위계, CTA, 카드 구조, 상세 섹션, 분위기를 제품 수준으로 다듬는다.

5
Superpowers Writing Plans

기술스택, UI 제약, API 흐름, 컴포넌트 후보를 구현 계획으로 변환한다.

6
Superpowers Git Worktrees

기능 단위를 안전하게 쪼개고 merge, env, install 규칙까지 고정한다.

7
Codex 분석 · 구현

현재 코드베이스를 읽고 최소 수정 원칙 아래 실제 구현과 검증까지 진행한다.

역할 분담

디자인 창작과 구현을 분리하고, 각 도구가 가장 잘하는 지점만 맡긴다.

Codex

  • 문제 정의 인터뷰
  • 코드베이스 분석
  • 구현 계획 고정
  • 실제 코드 작성
  • 테스트와 검증

Superpowers

  • 브레인스토밍
  • 구현 계획 문서화
  • 태스크 분해
  • worktree 전략 정리

Stitch

  • 기준 화면 시안 생성
  • 레이아웃과 섹션 기준점 생성
  • 가능하면 퍼블리시 결과물 확보

Claude

  • Stitch 결과를 제품 구조로 정제
  • 정보 위계와 카드 구조 정리
  • cheap한 AI 느낌 제거
  • 최종 UI 완성도 보정

단계별 운영 방식

기술스택과 UI 제약은 너무 일찍 고정하지 않고, Writing Plans에서 본격 반영한 뒤 구현 단계에서 유지한다.

1단계. Codex /plan 인터뷰

문제 정의, 사용자, 핵심 가치, MVP 범위와 결과 단위를 먼저 확정한다.

  • 기능 목록과 화면 구조는 아직 만들지 않는다.
  • 기술스택은 깊게 논의하지 않고 제품 제약만 말한다.
  • 추론 강도: 심화추론
2단계. Superpowers Brainstorming

빠진 요구사항과 UX 리스크, edge case, 상태 처리 구멍을 찾는다.

  • 기술보다 사용자 흐름과 서비스 신뢰성에 집중한다.
  • 추천·탐색·입력·저장·관리 흐름을 점검한다.
  • 추론 강도: 심화추론
3단계. Stitch 기준 디자인

주요 화면의 기준 디자인을 잡아 이후 구현의 시각적 기준점으로 삼는다.

  • 구현 기준점으로 사용할 수 있을 만큼 완성도 있게 만든다.
  • 첫인상이 중요한 서비스일수록 효과가 크다.
  • 추론 강도: 보통
4단계. Claude 구조 정제

Stitch 결과를 구현 가능한 제품 구조로 다듬고 정보 위계를 정교하게 만든다.

  • 새 디자인 창작보다 제품화에 초점을 둔다.
  • 여행·커머스·브랜드 서비스에서 특히 중요하다.
  • 추론 강도: 보통 ~ 약간 심화
5단계. Superpowers Writing Plans

기술 전제, UI 제약, API 흐름, 컴포넌트 구조를 구현 계획으로 바꾼다.

  • 기술스택을 여기서 처음 본격적으로 확정한다.
  • 브라우저 기본 input, modal 지양 같은 제약도 여기서 잠근다.
  • 추론 강도: 보통, 구현 전략 판단은 약간 심화
6단계. Superpowers Git Worktrees

기능 단위 분리와 운영 규칙을 함께 설계해 병렬 작업 리스크를 줄인다.

  • foundation과 feature 작업을 분리할지 판단한다.
  • gitignore, env, install, merge 규칙까지 함께 정리한다.
  • 추론 강도: 보통
7단계. Codex 코드베이스 분석

Writing Plans와 Stitch·Claude 구조를 현재 저장소 구조에 자연스럽게 매핑한다.

  • 관련 파일 후보, 라우팅 구조, 진입점, 위험한 영향 범위를 찾는다.
  • 기술 선호를 다시 길게 설명하지 않는다.
  • 추론 강도: 심화추론
8~10단계. 구현 계획 고정 · 구현 · 최종 정제

최소 수정 원칙으로 구현하고, 마지막엔 cheap함만 걷어낸 세련된 완성도로 다듬는다.

  • 공통 컴포넌트 재사용성과 정보 구조를 함께 챙긴다.
  • boring하거나 촌스러운 방향으로 단순화하지 않는다.
  • 추론 강도: 대체로 보통

범용 프롬프트 템플릿

프로젝트마다 아래 템플릿에서 프로젝트 설명, 사용자, 핵심 흐름, 기술스택, UI 제약, 데이터/API 축만 교체해서 쓰면 된다.

1. Codex /plan

/plan

지금은 구현 전에 문제 정의와 MVP 범위를 좁히는 단계다.
바로 기능 목록, 화면 구조, 구현안으로 넘어가지 말고 먼저 나를 인터뷰해줘.

규칙:
- 한 번에 최대 5개 질문만 할 것
- 내 답변 전에는 계획 확정이나 구현으로 넘어가지 말 것
- 내가 준 정보는 가설로 보고 검증 질문을 할 것
- 목표는 핵심 문제, 주요 사용자, MVP 흐름, 범위/비범위를 명확히 하는 것

8. Codex 구현

이제 위 계획대로 MVP를 구현해줘.

규칙:
- 관련 파일만 수정
- 공통 컴포넌트는 재사용 가능하게 만들 것
- 구현 후 lint / typecheck / test 가능한 범위에서 실행

추가 제약:
- 브라우저 기본 input처럼 보이는 UI는 피할 것
- modal/dialog 중심 흐름을 사용하지 말 것

핵심 운영 원칙

기술스택, Stitch 참조 시점, UI 제약, AI 슬롭 대응 방식을 한 번에 정리한다.

기술스택과 Stitch

기술스택 1단계에서는 제품 제약만, 5단계 Writing Plans에서 본격 반영한다.
Stitch 참조 3단계에서 생성하고 5단계, 7단계, 9단계에서 구현 기준점으로 사용한다.
UI 제약 브라우저 기본 input, modal 중심 흐름 지양 규칙은 Writing Plans에서 확정한다.

AI 슬롭 대신 정제

피해야 할 표현 AI 슬롭 제거, 과한 요소 제거, 현실적으로만 수정 같은 지시는 디자인을 죽이기 쉽다.
권장 방향 감성 유지, premium 무드 유지, cheap한 느낌만 제거, 위계와 spacing만 정교하게 다듬는다.
핵심 문장 슬롭 제거보다 세련되게 정제가 더 맞다.

최종 압축 버전

새 프로젝트를 시작할 때 빠르게 복기할 수 있는 한 줄 흐름이다.

1. Codex /plan 인터뷰
2. Superpowers Brainstorming
3. Stitch로 기준 디자인/퍼블리시 결과물 만들기
4. Claude로 구조/톤/정보 위계 보정
5. Superpowers Writing Plans
6. Superpowers Git Worktrees
7. Codex 코드베이스 분석
8. Codex 구현 계획 고정
9. Codex 실제 구현
10. Claude로 최종 UI 정제
Stitch가 먼저 화면 기준점을 만들고, Claude가 그걸 제품 구조로 다듬고, Superpowers가 실행 계획으로 바꾸고, Codex가 구현하는 흐름을 반복 가능한 범용 프레임워크로 사용하는 것이 이 문서의 핵심이다.
